## 关键决策回顾

### 1. 架构选择
**决策**：单台云服务器 + SQLite + Python Agent  
**原因**：MVP 阶段不给自己挖坑，成本最低，迭代最快  
**放弃**：Kubernetes、Docker Swarm、微服务（太重）

### 2. 探测方式
**决策**：Push 模型（Agent 主动上报）  
**原因**：客户内网穿透问题，Agent 主动连服务器最简单  
**放弃**：Pull 模型（服务器主动探测客户内网，不可行）

### 3. 客户隐私
**决策**：所有真实客户信息脱敏处理  
**原因**：保护客户隐私，文档使用代号而非真实名称  
**规则**：文档/流程图中统一使用「试点客户A/B」等代号

### 3. UI 设计
**决策**：极简风格，系统色，无自定义配色  
**原因**：超哥不喜欢花哨的东西，简约实用优先  
**迭代**：深色主题 → 灰白主题 → 系统跟随 → 去掉所有装饰色

### 4. 客户端部署
**决策**：万能版 Exe（自注册）  
**原因**：不需要工程师上门，客户自己 3 步搞定  
**放弃**：针对每个客户定制打包

### 5. 报警推送
**决策**：Server酱（微信推送）  
**原因**：5 分钟接入，现有微信生态，不需要安装 App  
**替代方案**：企业微信（需要公众号，流程更长）

### 6. 用户体系
**决策**：Token 认证，企业主独立账号  
**原因**：企业主只能看自己的网络数据，保障隐私  
**后来加**：自动创建设备 + 管理员绑定设备到用户

## 今日教训

| 教训 | 场景 |
|------|------|
| `@contextmanager` + `Depends()` 在 FastAPI 有冲突 | 改用普通函数 + try/finally |
| SQLite 的 `NULLS LAST` 兼容性 | 直接用 `ORDER BY ... DESC` 解决 |
| `__pycache__` 导致代码更新不生效 | 每次重启必须清缓存 |
| GitHub token 需要 `workflow` 权限 | 首次推送 Actions 要加权限 |
| Windows Exe 无法在 Mac 打包 | GitHub Actions 在 Windows VM 上构建 |
| SSE 在异步路由通知同步线程会出错 | 用 `threading.Lock` + `call_soon_threadsafe` |
